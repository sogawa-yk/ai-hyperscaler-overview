# 第2章 アプリ／エージェント実行基盤 ― マネージドK8s・サーバーレス・レジストリ ― 仕様書

## 章の概要

**章の目的**: 実行基盤を「マネージドAIを叩くアプリ／エージェントのホスト」として最小限に切り、マネージドK8s・サーバーレスコンテナ・レジストリの軸で4社をプロットする。AIのランタイムそのものではなく「ホスト」としての位置づけを徹底する。

**目標ページ数**: 20p

**所属する部**: 第II部 土台となる部品

## 前提知識

この章を読むために必要な前提:
- 序章の方法論
- Kubernetes・コンテナの基本（読者の得意領域）
- 第1章のワークロードID（ホストが「誰として」AIを叩くか）

## 節の構成

### 2.1 軸の導入 ― 実行基盤を「管理の手放し度」で切る

**概要**: コンテナの実行基盤を「自分で管理する量（ノード管理あり/なし、スケールの粒度）」という軸で切る。本書では実行基盤を「AIアプリ／エージェントのホスト」と位置づけ、深入りしない方針を述べる。

**図表**:
- 図2.1: 管理の手放し度の軸（マネージドK8s〜サーバーレス）

**キーポイント**:
- 軸は「どこまで事業者に任せるか」
- AIのホストとして見るので最小限に扱う

### 2.2 4社プロット ― K8s・サーバーレス・レジストリ

**概要**: マネージドK8s（EKS/AKS/GKE/OKE）、サーバーレスコンテナ（Fargate/Container Apps/Cloud Run/Container Instances）、レジストリ（ECR/ACR/Artifact Registry/OCIR）を軸上に置く。

**図表**:
- 表2.1: 実行基盤の4社プロット（K8s/サーバーレス/レジストリ）

**キーポイント**:
- 各社ともK8sとサーバーレスの二層構造を持つ
- レジストリはほぼ横並び

### 2.3 対訳（他社→OCI）

**概要**: EKS ≒ OKE、Fargate ≒ Container Instances、ECR ≒ OCIR 等を記号付きで示す。サーバーレス領域の細部の差（スケールゼロ等）を △ で扱う。

**図表**:
- 表2.2: 実行基盤の対訳表

**キーポイント**:
- ほぼ ≒ で対応する成熟領域
- 差はスケール挙動や統合機能の細部

### 2.4 両方向ギャップとSWOTスライス

**概要**: GKE Autopilot のような運用自動化、Cloud Run の成熟度など他社の強みと、OCIの位置を整理する。OCIの弱み（サーバーレスコンテナの機能差等）を明示する。

**図表**:
- 表2.3: 実行基盤の両方向ギャップとSWOTスライス

**キーポイント**:
- 成熟領域ゆえギャップは小さいが細部に差
- OCIの弱みを正直に書く

### 2.5 ホストとしての実行基盤 ― マネージドAIへの接続点

**概要**: 実行基盤がマネージドAI（第4章）をどう叩くか、ワークロードID（第1章）をどう使うかという接続点を整理する。エージェント実行のホストとしての観点を述べる。

**図表**:
- 図2.2: アプリ／エージェントがマネージドAIを叩く構図

**キーポイント**:
- 実行基盤は「叩く側」、AIは「叩かれる側」
- IDと伝播がここで効く

### 2.6 新顔の分類手順と確認日

**概要**: 新しい実行基盤製品を「管理の手放し度」と「AIホストとしての適性」で分類する手順を示す。章末に確認日を記す。
