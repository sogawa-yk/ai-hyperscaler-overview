## 節の構成

### 序.1 なぜ「製品の辞典」ではなく「地図」なのか

**概要**: クラウドAIの製品は爆発的に増え、しかも急速に陳腐化する。製品名を覚える学習は賞味期限が短い。本書は製品ではなく「軸（役割）」を覚える地図づくりを目的とすることを宣言する。

**図表**:
- 図序.1: 製品の網羅（辞典）と軸の理解（地図）の対比

**キーポイント**:
- 覚えるのは軸であって製品名ではない
- 地図があれば未知の新製品を自力で置ける

### 序.2 軸という考え方 ― 役割で世界を切る

**概要**: 「軸」とは領域を見通す座標（役割）である。製品は軸上の点（プロット）にすぎない。各領域章で同じ5つのレンズ（対訳・両方向ギャップ・実現方式・SWOTスライス・分類手順）を当てることを説明する。

**図表**:
- 図序.2: 1本の軸の上に4社の製品をプロットした模式図

**キーポイント**:
- 軸は不変、プロットは更新対象
- 同じレンズを全領域に当てるから比較できる

### 序.3 OCIを原点に置く理由

**概要**: すべてを相対座標で語るには基準点が要る。本書は読者の母艦であるOCIを原点に置き、「OCIではここ、他社はここ」と記述する。対訳記号（≒／△／なし）を導入する。

**図表**:
- 表序.1: 対訳記号（≒／△／なし）の定義と使い分け

**キーポイント**:
- 原点があるから他社製品を即座に位置づけられる
- 原点をOCIにするのは読者の利便のためで、優劣の主張ではない

### 序.4 スナップショットと軸の分離 ― 陳腐化への備え

**概要**: モデル数・GA/プレビュー・リージョン・ブランド改称は陳腐化する。各章に確認日を付し、不変の軸と更新対象のプロットを分離する運用を説明する。出典主義と「要確認」の明示も述べる。

**図表**:
- 図序.3: 軸（不変）とスナップショット（更新対象）の二層構造

**キーポイント**:
- 確認日付きで事実を扱う
- 確認できない主張は断定しない

### 序.5 本書の使い方と各章の読み方

**概要**: 領域章の固定小節構成（軸の導入→4社プロット→対訳→両方向ギャップ→SWOTスライス→新顔の分類手順→確認日）、ケイパビリティ・カード、読書経路の選び方を案内する。

**図表**:
- 図序.4: 領域章の固定フォーマットと読み進め方

**キーポイント**:
- 各領域章は自己完結的に読める
- 論点が立つ要件はカードで深掘りする
