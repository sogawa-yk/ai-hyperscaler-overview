# 書籍企画書 (Book Plan)

## 書籍概要

### タイトル
**AIワークロード クラウド4社 知識地図** ― OCIを原点に、AWS・Azure・Google Cloudを相対座標で読み解く

### 形態
- 判型: B5
- 想定ページ数: 200ページ
- 配布形態: 技術書（自己学習・参照教材）

### コンセプト
- **製品ではなく軸を教える**: 各領域でまず座標軸（役割）を立て、4社（AWS / Azure / Google Cloud / OCI）の製品はその軸上の点として扱う。読者が記憶するのは陳腐化しない「軸」であり、個々の製品名ではない。
- **OCIを原点に置く**: すべての軸を「OCIではここ、他社はここ」という相対座標で語る。他社の任意の製品が「OCIでは何に相当するか」を即答できる地図を読者の頭に作る。
- **スナップショット＋更新前提の正直な記述**: 製品名・モデル数・GA/プレビュー・リージョンは陳腐化する。各章に確認日を付し、不変の「軸」と更新対象の「プロット」を明確に分離する。マーケ的優位主張を避け、OCIの弱みを含めて出典主義で書く。

### ビジョン
本書は、AWS・Azure・Google Cloud・OCIの4社について、AIワークロードに必要な部品を横断的に並べ、読み終えたとき読者の頭の中に「クラウドAIの知識地図」ができあがることを唯一の目的とする。個々の製品スペックを網羅する辞典ではなく、「どんな軸で世界を切ると4社が見通せるか」「新しい製品に出会ったとき自分で地図のどこに置けるか」という"読み方・並べ方"を読者に渡す。読者は読了後、未知の新製品を自力で地図の正しい位置に置けるようになる——これは賞味期限のない能力である。

## 対象読者

### プライマリーペルソナ
- **職種・役割**: クラウドベンダーのプリセールス／ソリューションアーキテクト。全業種のAI系新規ワークロード案件を担当し、提案・技術検証の上流を担う。
- **技術レベル・経験**: クラウドネイティブ基盤（Kubernetes / OKE、IaC、観測性）に強い実務経験を持つ。アーキテクチャ全体像を俯瞰して語ることに慣れている。
- **前提知識**: 1つのクラウド（特にOCI）の主要サービスは実務レベルで理解している。コンテナ・ネットワーク・IAMの基本概念を把握している。
- **課題**: 他社（AWS / Azure / GCP）のAI関連サービスを「OCIの何に相当するか」という対応関係で素早く把握したい。特にデータ領域（DB設計・運用、データ基盤、DB組み込みAI）に自認のギャップがあり、ここを埋めたい。マーケ的フレーミングに懐疑的で、出典・第一原理ベースの正直な記述を求める。

### 前提知識
読者が本書を読む前に理解していることを前提とする:
- クラウドコンピューティングの基本概念（IaaS / PaaS / SaaS、リージョン／可用性、IAMの基礎）
- コンテナとKubernetes（マネージドK8sの利用経験）
- いずれか1社のクラウド（特にOCI）の主要サービスの実務知識
- LLM・生成AIの基本用語（プロンプト、トークン、埋め込み、RAGの概念）

### 前提としない知識
本書で説明する:
- 4社それぞれのAI関連サービスの役割と相互の対応関係
- DB組み込みAI（DB内ベクトル検索、SQLからのLLM呼び出し、NL→SQL、データ層のきめ細かい認可）
- クラウド間のID委譲・トークン交換・ID伝播の仕組み
- AIワークロードに固有の観測性・ガバナンスの考え方

## 到達目標

読了後、読者が以下をできるようになること:

1. 他社の任意の製品について「OCIでは何に相当するか」を即答できる（対訳）
2. 「他社にあってOCIにないもの」「OCIにあって他社にないもの」を双方向に説明できる
3. AIワークロードの必要部品を分解し、各社の実現方式とOCIの実現方式を対応づけられる
4. AIワークロードにおける各社の強み・弱み（OCIの弱みを含む）を出典のある範囲で語れる
5. 未知の新製品を、自力で地図の正しい位置（軸上の点）に置ける

## 執筆方針

- **役割（軸）優先**: すべて「役割（軸）→製品」の順で書く。製品名から入らない。各領域章は固定の小節構成（軸の導入→4社プロット→対訳→両方向ギャップ→SWOTスライス→新顔の分類手順→確認日）を守る。
- **出典主義・正直さ**: 各事実は一次情報（各社公式ドキュメント／ブログ、リリースノート）に辿れること。アナリスト記事・二次記事は補助。OCIの弱みを省略せず、優位主張は出典のある範囲に限定する。確認できない主張は断定せず「要確認」と明示する。
- **陳腐化の明示**: モデル数・GA/プレビュー・リージョン・ブランド改称（例: Azure AI Studio→AI Foundry、Oracle Database 23ai→26ai）は確認日付きで記す。不変の軸と更新対象のプロットを分離する。
- **再現可能な分類手順**: 各領域章末に「この軸で未知の新製品をどう分類するか」の再現手順を必ず置き、読者が自走で地図を更新できるようにする。
- **ケイパビリティ・カード**: 論点が立つ要件（委譲・ID伝播、データ層認可など）は固定テンプレート（課題／OCIでの実現／他社での実現／差分の見立て／確認日）でカード化する。

## スコープ外

明示的にスコープ外とする項目:
- ハンズオン／デプロイ手順（本書は「読んで理解する横断リファレンス」。唯一の例外は終章の公式RA読解で、これも構築は行わない）
- 製品の網羅的な全機能リファレンス化（軸の理解に必要な範囲に絞る）
- アプリケーション実装コード・サンプルアプリの開発（コード例はSQL／設定断片の最小限のみ）
- 価格の詳細比較・SLAの逐条比較（軸の理解に必要な範囲を超える定量比較はしない）
- AIモデルそのものの内部アルゴリズム解説（Transformerの理論等）
