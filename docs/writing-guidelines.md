# 執筆ガイドライン (Writing Guidelines)

本書は「軸を教える・OCIを原点に置く・出典主義で正直に書く」を貫く横断リファレンスである。以下のルールを全章の執筆・レビューの基準とする。

## 文体ルール

### 基本文体
- **文体**: 常体（である調）
- **一人称**: 使用しない（「本書では」「ここでは」等を使用）
- **読者への語りかけ**: 最小限（「〜してみよう」等は避ける）
- **トーン**: 技術的に率直、アーキテクチャ中心、簡潔。マーケティング的な語彙を避ける（後述の禁止事項を参照）

### 文の長さ
- **推奨**: 1文60字以内
- **上限**: 1文80字
- **対処**: 80字を超える場合は分割する

### 段落の長さ
- **推奨**: 1段落3-5文
- **上限**: 1段落7文
- **対処**: 長い場合は段落を分割し、適切な見出しを付ける

### 接続詞
- 多用を避ける
- 特に「また」「そして」の連続使用は禁止
- 因果関係は「そのため」「したがって」を優先

## 表記ルール

### 技術用語
- **初出時**: 日本語（英語）の形式
  - 例: 委譲（Delegation）、ID伝播（Identity Propagation）
- **2回目以降**: 日本語のみ（定着した日本語訳がある場合）
- **略語**: 初出時に正式名称を併記
  - 例: RAG（Retrieval-Augmented Generation、検索拡張生成）
- **製品名・サービス名**: 公式表記をそのまま使う（無理に訳さない）
  - 例: Amazon Bedrock、Microsoft Foundry、Vertex AI、OCI Generative AI
- 用語の統一基準は `docs/glossary.md` に従う

### 4社の表記（最重要・厳守）
- クラウド4社の表記は以下に統一する:
  - **AWS**（Amazon Web Services）
  - **Azure**（Microsoft Azure）
  - **Google Cloud**（GCP と略さない。略す場合のみ初出で併記）
  - **OCI**（Oracle Cloud Infrastructure）
- 4社を列挙する順序は原則 **AWS → Azure → Google Cloud → OCI** で固定する（OCIを原点＝基準点として末尾に置く）
- 表のプロットも同じ列順を守る

### 対訳・差分の記号
他社製品のOCI相当を示す対訳では以下の記号を統一して使う:
- **≒**: ほぼ相当する（役割・実現方式が対応）
- **△**: 部分的に相当する（一部の機能のみ、または実現方式が異なる）
- **なし**: OCIに直接相当するものがない
- SWOTスライスでは各社の強み（S）・弱み（W）・機会（O）・脅威（T）を明示し、OCIの弱み（W）を必ず含める

### 確認日（スナップショット）の表記
- 陳腐化しやすい事実（モデル数・GA/プレビュー状態・リージョン・ブランド改称）には確認日を付す
- 章末に「確認日: YYYY-MM-DD」を記し、特に陳腐化しやすい項目は注記する
- 本文中で個別に注記する場合は「（確認日: 2026-06-09）」の形式
- 不変の「軸」と更新対象の「プロット（製品名等）」を文章上も区別する

### 数字
- **1桁**: 漢数字（一つ、二つ）
- **2桁以上**: アラビア数字（10個、100件）
- **技術的な数値**: 常にアラビア数字（4社、RFC 8693、ポート443）

### 記号
- **括弧**: 全角括弧（）を基本とする。英語・技術用語は半角()
- **コロン**: 説明の区切りには全角「：」
- **ピリオド・カンマ**: 日本語文中では「。」「、」

### 見出し
- **章タイトル**: `# 第N章 タイトル ― サブタイトル`（序章・統合章・終章は番号なし）
- **節タイトル**: `## N.X タイトル`
- **小見出し**: `### タイトル`
- **ダーシ**: 全角ダッシュ「―」を使用（ハイフンではない）

## 図表ルール

### 図表番号
- **図**: 図X.Y（X=章番号、Y=章内連番）。序章・統合章・終章は「図序.Y」「図統.Y」「図終.Y」とする
- **表**: 表X.Y（対訳表・4社プロット表が中心）
- **コード例**: リストX.Y（SQL・設定断片を引用する場合）

### キャプション
- 図表の直下に配置
- 簡潔に内容を説明する（1文）
- 例: 「表4.1: マネージドAIの4社プロット」

### 本文中の参照
- 「図4.1に示すように」「表4.2を参照」の形式
- 図表は必ず本文中で参照する（参照されない図表は不要）
- **各節に最低1つの図表を配置する**

### Mermaid記法
- ノード名は日本語
- 矢印にラベルを付ける
- シンプルに保つ（ノード数は10個以内を推奨）
- 配色ルールは `figure-design` スキルに従う（ダークテーマ前提・白文字）

## コード例ルール

本書はハンズオン書ではないため、コード例は最小限に留める。

### 言語
- 例示が必要な場合に限り、以下を使う:
  - **SQL**: Select AI / NL→SQL / DB内ベクトル検索の例（` ```sql `）
  - **設定・ポリシー断片**: JSON / YAML（IAMポリシー、ガードレール設定等）
  - **プロトコル例**: OAuth 2.0 Token Exchange（RFC 8693）のリクエスト/レスポンス
- アプリケーション実装コードは扱わない
- 概念説明のための疑似コードを使う場合は、コメントで「疑似コード」と明示する

### 形式
- コードブロックには言語名を指定する
- 必ずコメントを付ける
- 概念理解に必要な最小限に絞る

### インデント
- SQL/JSON/YAML とも2スペースを基本とする（言語慣習に従う）

## 引用・参照ルール

本書は出典主義を品質の核とする。事実は一次情報に辿れる形で書く。

### 情報ソースが必要な記述

以下の記述には**必ず情報ソースを付ける**:
- 製品の存在・機能・対応状況（GA/プレビュー、リージョン、モデル対応等）に関する具体的主張
- 「他社にあってOCIにない」「OCIにあって他社にない」というギャップの断定
- ブランド改称・バージョンの事実（例: Azure AI Studio→AI Foundry）
- 「一般的には〜」「〜が主流」等の一般化した主張、統計・市場シェア・事例

### 情報ソースの優先順位
1. **一次情報**: 各社公式ドキュメント、公式ブログ、リリースノート（最優先）
2. **補助**: アナリストレポート、二次記事（一次情報を補う場合のみ）

### 情報ソース不要な記述
- 本書独自の「軸」の定義（「この領域はこういう役割で切る」等）
- 論理的な帰結（推論）
- 本書の他章への参照

### 確認できない主張の扱い
- 一次情報で確認できない主張は**断定せず「要確認」と明示する**
- マーケティング的優位主張は出典のある範囲に限定し、前提（例: 「規模自体は他社も大規模」）を併記する

### 情報ソースの書き方
本文中では脚注形式で記載する:

```markdown
Azure の基盤モデル基盤は Microsoft Foundry に改称された[^1]。

[^1]: Microsoft "Azure AI Foundry documentation" (2026), https://learn.microsoft.com/...
      （確認日: 2026-06-09）
```

各章末に参考文献セクションを設ける:

```markdown
## 参考文献

- [著者/組織名] "[タイトル]" ([年]), [URL]（確認日: YYYY-MM-DD）
```

### 章間の参照
- 「第X章で述べたように」「X.Y節を参照」の形式
- 前方参照（まだ説明していない内容への参照）は最小限にする

## ケイパビリティ・カードのフォーマット

論点が立つ要件（委譲・ID伝播、データ層認可など）は、領域章内で以下の固定テンプレートでカード化する:

```markdown
### ケイパビリティ・カード: [要件/能力名]

- **課題**: 解決したい問題（なぜこの要件が要るか）
- **OCIでの実現**: 製品・プロトコル・仕組み
- **他社での実現**: AWS / Azure / Google Cloud が同じ要件をどう満たすか
- **差分の見立て**: どこがOCIの強みで、どこは横並びか（断定は出典のある範囲で）
- **確認日**: YYYY-MM-DD
```

## 領域章の固定小節構成

第1〜6章は以下の順序を厳守する（concept §8）:

1. 軸の導入
2. 4社プロット（表）
3. 対訳（他社→OCI、`≒ / △ / なし`）
4. 両方向ギャップ
5. SWOTスライス（OCIの弱みを必ず含む）
6. 新顔の分類手順
7. 確認日（章末）

## 理解度チェック問題

各章末に理解度チェック問題を置く。フォーマットは `review` / `quiz-format-checker` の基準に従う。本書の性質上、「他社製品Xを地図のどこに置くか」「OCI相当は何か」を問う設問を中心とする。

## 禁止事項

- 「〜と思います」「〜でしょう」等の推量表現（技術書では断定する。ただし不確実な事実は「要確認」と明示）
- 「簡単に」「すぐに」等の主観的な難易度表現
- 「最近では」「近年」等の曖昧な時間表現（具体的な年・確認日を記載）
- 読者を見下すような表現（「当然ですが」「言うまでもなく」）
- 未定義の略語の使用
- マーケティング的優位主張（「業界最高」「唯一の」等。出典のある事実に置き換える）
- OCIの弱みの省略（SWOTスライスでは必ず明示する）
