# 書籍コンセプト案（執筆スペック）

**仮タイトル**：AIワークロード クラウド4社 知識地図 — OCIを原点に、他社標準を読み解く

> 本ドキュメントの位置づけ：これは**書籍の執筆仕様（スペック）**であり、書籍本体ではない。実際の執筆は別のコーディングエージェントが本スペックに従って行う。
> 既存の事実シード：`ai-workload-cloud-knowledge-map.md`（同一会話で作成済み）を「現時点スナップショットの事実の種」として参照する。
> 基準日：2026-06-09（各章で再確認・上書きする前提）

## 1. コンセプト概要（エレベーターピッチ）

AWS / Azure / Google Cloud / OCI の4社について、AIワークロードに必要な部品を横断的に並べ、**読み終えたとき読者の頭の中に「クラウドAIの知識地図」ができあがる**ことを唯一の目的とする書籍。個々の製品スペックの網羅辞典ではなく、**(1) どんな軸で世界を切ると4社が見通せるか、(2) 新しい製品に出会ったとき自分で地図のどこに置けるか**という"読み方・並べ方"を読者に渡す。OCIを地図の原点に置き、すべてを相対座標で語る。

## 2. 目的・ゴール

読了後に読者ができるようになること：
- 他社の任意の製品について「OCIでは何に相当するか」を即答できる
- 「他社にあってOCIにないもの」「OCIにあって他社にないもの」を説明できる
- AIワークロードの部品を分解し、各社の実現方式とOCIの実現方式を対応づけられる
- AIワークロードにおける各社の強み・弱み（OCIの弱みを含む）を語れる
- **未知の新製品を、自力で地図の正しい位置に置ける**（賞味期限のない能力）

## 3. 想定読者

- 一次読者：本人（Oracleプリセールス、Pursuitチームへの移行中、全業種のAI系新規ワークロード案件担当）。自分用の学習・参照教材。
- 前提知識：クラウドネイティブ基盤（Kubernetes/OKE、IaC、観測性）に強い。データ領域（DB設計・運用、データ基盤）に自認のギャップ → 本書のデータ側AI章はギャップ埋めを兼ねる。
- トーン適性：マーケ的フレーミングへの強い懐疑。出典・第一原理ベースの正直な記述を好む。

## 4. 設計原則（執筆時の絶対ルール）

1. **製品ではなく軸を教える**：各章はまず座標軸（役割）を立て、4社の製品はその軸上の点として扱う。読者が覚えるのは軸。
2. **OCIを原点に置く**：すべての軸を「OCIではここ、他社はここ」と相対座標で語る。
3. **各章末に分類手順を残す**：「この軸で新顔をどう分類するか」の再現可能な手順を必ず置く。
4. **スナップショット＋更新前提**：製品名・モデル数・GA/プレビュー・リージョンは陳腐化する。各章に「確認日」を付し、軸（不変）とプロット（更新対象）を明確に分離する。
5. **正直・出典主義**：マーケ的優位主張を避ける。OCIの弱みを明示する。各事実は出典を辿れる形にし、確認できない主張は断定しない。

## 5. スコープ

**カバーする部品領域**（必須）：IDaaS／コンテナ基盤／オブジェクトストレージ／AI系サービス（マネージド重心）／**データ側AI（DB組み込み）**。必要に応じて o11y・ガバナンス。

**重心の置き方**：
- AIはマネージドサービス側に重心（Bedrock / Foundry / Vertex / OCI Generative AI 系）。
- コンテナ基盤は「AIのランタイム」ではなく「マネージドAIを叩くアプリ/エージェントのホスト」として最小限に扱う。
- **データ側AI（DB組み込み）は第一級の領域**として独立させる（Oracleの強みの主戦場＋読者のギャップ埋め）。

**非スコープ**：
- ハンズオン／デプロイ手順は含めない（読んで理解する横断リファレンス）。唯一の例外は終章の「公式RA読み比べ」（構築せず読解のみ）。
- 製品の網羅的な全機能リファレンス化はしない（軸に必要な範囲に絞る）。

## 6. 必須コンテンツ（本書が必ず満たす5点）

1. 他社の各製品はOCIでは何に相当するか（対訳）
2. 他社にあってOCIにないもの
3. OCIにあって他社にないもの
4. AIワークロードの必要部品は何で、他社はどう実現し、OCIではどう実現するか
5. AIワークロードにおける各社の強み・弱み（他社の強み／OCIの弱み／OCIの強み／他社の弱み）

これら5点は、`ai-workload-cloud-knowledge-map.md` の Part 1〜5 に現時点スナップショットが入っている。本書はこれを領域ごとに分解・肉付けして提示する。

## 7. 全体構成（章立て）

breadth-first（全体像優先）。領域ごとに章を立て、各章内で5つのレンズ（対訳・両方向ギャップ・実現方式・SWOTスライス）を当てる。最後に全体を統合する。

- **序章｜地図の読み方**：設計原則、軸という考え方、OCIを原点に置く理由、本書の使い方とスナップショットの扱い。
- **第1章｜アイデンティティ／認可**：ワークフォース／CIAM／ワークロードID、クラウド間フェデレーション、**委譲・ID伝播（Entra OBO ↔ OCI Token Exchange + Identity Propagation Trust）**、エージェントID（Entra Agent ID と OCI の組み立て）。
- **第2章｜アプリ/エージェント実行基盤**：マネージドK8s（EKS/AKS/GKE/OKE）、サーバーレスコンテナ、レジストリ。「マネージドAIのホスト」としての位置づけ。
- **第3章｜オブジェクトストレージ**：S3/Blob/GCS/OCI Object Storage、RAGの一次データ置き場、レイクハウス／テーブルフォーマットへの接続。
- **第4章｜マネージドAI**：基盤モデル/推論、エージェント基盤、マネージドRAG、ガードレール、MLプラットフォーム、MCP対応。
- **第5章｜データ側AI（DB組み込み）**：DB内ベクトル、SQLからのLLM／NL→SQL（Select AI 等）、**データ層のきめ細かい認可（Deep Data Security ↔ 他社のTrusted Identity Propagation＋RLS/CLS）**、ハイブリッド検索、レイクハウス横断ポリシー。
- **第6章｜o11y／ガバナンス**：クラウドネイティブo11y、LLM/エージェント特化o11y（OTel GenAI semconv / Langfuse はクラウド非依存の共通層）、AIゲートウェイ、行動監査（Agent 365 ↔ Deep Data Securityの監査）。
- **統合章｜地図の全体像**：全領域を横断した（A）他社→OCI対訳辞書、（B）他社にありOCIにない一覧、（C）OCIにあり他社にない一覧、（D）AIワークロードSWOT、を集約。
- **終章｜公式RA読み比べ（capstone・構築なし）**：同一題材（マネージドRAGアシスタント）の各社公式リファレンスアーキテクチャ（AWS SRA GenAI／Azure AI Landing Zones／GCP Cloud Architecture Center／OCI Architecture Center）を並べ、全ボックスを本書の軸の語彙にマッピングする読解。

## 8. 章の標準フォーマット（各領域章で必ずこの順）

各領域章（第1〜6章）は以下の小節構成を固定する：

1. **軸の導入**：この領域を見通すための座標軸（役割）を立てる。
2. **4社プロット**：各社の製品を軸上に配置（表）。製品名＋1フレーズの役割説明に留める。
3. **対訳（他社→OCI）**：代表製品のOCI相当を `≒ / △ / なし` で示す。
4. **両方向ギャップ**：この領域で「他社にありOCIにない」「OCI固有」を各1〜数項目。
5. **SWOTスライス**：この領域に限った各社の強み/弱み（OCIの弱みを必ず含む）。
6. **新顔の分類手順**：この軸で未知の新製品をどう分類するかの再現手順。
7. **確認日**：章末に snapshot 基準日と、特に陳腐化しやすい項目の注記。

## 9. ケイパビリティ・カードのテンプレート（領域章内の反復単位）

特に論点が立つ要件（例：委譲・ID伝播、データ層認可）はカード化する：

```
### [要件/能力名]
- 課題：解決したい問題（なぜこの要件が要るか）
- OCIでの実現：製品・プロトコル・仕組み
- 他社での実現：AWS / Azure / GCP が同じ要件をどう満たすか
- 差分の見立て：どこがOCIの強み／どこは単なる横並びか（断定は出典のある範囲で）
- 確認日：YYYY-MM-DD（陳腐化しやすい事実の注記）
```

シード済みカード例（`ai-workload-cloud-knowledge-map.md` 由来）：
- 「委譲・ID伝播」＝ Entra OBO ↔ OCI Token Exchange(RFC 8693) + Identity Propagation Trust（impersonationで代理元保持）→ さらにデータ層は Deep Data Security で貫通。
- 「データ層のきめ細かい認可」＝ Deep Data Security（DB内宣言ポリシー、行/列/セル、SQL生成元を問わず一貫）↔ 他社は Trusted Identity Propagation ＋ 各データサービスのRLS/CLS/ポリシータグの組み合わせ。

## 10. 品質基準

- **出典主義**：各事実は一次情報（各社公式ドキュメント/ブログ、リリースノート）に辿れること。アナリスト/二次記事は補助。
- **正直さ**：OCIの弱みを省略しない。優位主張は出典のある範囲に限定し、「規模自体は他社も大規模」のような前提も併記する（例：Supercluster は "ベアメタル＋価格性能" を主軸に語り、絶対規模の独自性は誇張しない）。
- **陳腐化の明示**：モデル数・GA/プレビュー・リージョン・ブランド改称（例：Azure AI Studio→AI Foundry→Microsoft Foundry、Oracle Database 23ai→26ai／"Oracle AI Database"）は確認日付きで記す。
- **トーン**：技術的に率直、アーキテクチャ中心、簡潔。マーケ語を避ける。
- **役割優先**：すべて「役割（軸）→製品」の順で書く。製品名から入らない。

## 11. コーディングエージェントへの指示（spec-driven authoring）

1. 本ドキュメントを**スペック**として扱い、章ごとに生成する。`ai-workload-cloud-knowledge-map.md` を**事実シード**として読み込む。
2. 各章の執筆前に、その章のスナップショット依存事実（製品名・モデル数・GA/プレビュー・リージョン・改称）を**Web検索で再確認**し、章末「確認日」を更新する。検証できなかった主張は断定せず「要確認」と明示する。
3. 出力は**1章＝1 Markdownファイル**（例：`ch01-identity.md`）。加えて front matter（タイトル・基準日・出典リスト）を持つ。
4. 各領域章は §8 の小節構成を固定で守る。論点が立つ要件は §9 のカード形式を使う。
5. §10 の品質基準（出典主義・正直さ・陳腐化明示・役割優先）を全章で適用する。
6. 統合章は各領域章の「対訳・ギャップ・SWOTスライス」を機械的に集約し、重複を排して全体表に再構成する。
7. 終章は構築を一切含めず、各社公式RAの**読解とマッピングのみ**を行う。
